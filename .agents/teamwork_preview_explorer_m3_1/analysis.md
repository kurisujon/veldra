# Milestone 3 Detailed Design Analysis: Backend Server Actions & Types

This document details the backend architectural design for Phase 4 Milestone 3 (Document Uploads and Management). It specifies the typescript types, validation schemas, and server action implementations required for case-centric document management in Veldra.

---

## 1. File Structure and Locations
To maintain Veldra's strict feature-based folder architecture, the following files will be added:
- **`src/features/documents/types.ts`**: TypeScript type definitions mapping database schemas and defining return signatures.
- **`src/features/documents/validation.ts`**: Zod validation schemas for validating file properties, file types, and UUIDs on the server.
- **`src/features/documents/actions.ts`**: Next.js Server Actions implementing the upload, deletion, and retrieval logic.

---

## 2. Interface Contracts & Database Alignments

### Database Alignment
The schema defined in `supabase/migrations/20260622000000_phase4_documents.sql` contains the following definition for `public.documents`:
- `id` (UUID, primary key)
- `case_id` (UUID, not null)
- `type` (TEXT, check constraint to `'PSABirth' | 'PSAMarriage' | 'TOR' | 'SF10' | 'Diploma'`)
- `file_path` (TEXT, unique, not null)
- `file_name` (TEXT, not null)
- `file_size` (INTEGER, not null)
- `mime_type` (TEXT, not null)
- `status` (TEXT, default `'uploaded'`)
- `uploaded_by` (UUID, defaults to `auth.uid()`)
- `created_at` / `updated_at` (TIMESTAMPTZ)

Our design leverages the autogenerated types from `src/types/database.ts` directly via the utility `Database['public']['Tables']['documents']['Row']`, enforcing 100% type alignment.

### Action Signatures
As per `PROJECT.md`, the actions return standard response objects:
- `uploadDocument(formData: FormData): Promise<UploadResponse>`
- `deleteDocument(documentId: string): Promise<DeleteResponse>`
- `getDocumentsByCase(caseId: string): Promise<GetDocumentsResponse>`

### 3.2 `src/features/documents/validation.ts`
```typescript
import { z } from 'zod'

// Allowed document types corresponding to DB constraints
export const AllowedDocumentTypes = ['PSABirth', 'PSAMarriage', 'TOR', 'SF10', 'Diploma'] as const;

export const UploadDocumentSchema = z.object({
  caseId: z.string().uuid({ message: 'Invalid Case ID format' }),
  type: z.enum(AllowedDocumentTypes, {
    errorMap: () => ({ message: 'Invalid document type' }),
  }),
  file: z
    .custom<File>((val) => val instanceof File, { message: 'File is required' })
    .refine((file) => file.size > 0, { message: 'File cannot be empty' })
    .refine(
      (file) => ['application/pdf', 'image/jpeg', 'image/png'].includes(file.type),
      { message: 'Only PDF, JPEG, and PNG formats are supported' }
    )
    .refine(
      (file) => file.size <= 10 * 1024 * 1024, // 10MB limit
      { message: 'File size must not exceed 10MB' }
    ),
})

export const DeleteDocumentSchema = z.object({
  documentId: z.string().uuid({ message: 'Invalid Document ID format' }),
})

export const GetDocumentsSchema = z.object({
  caseId: z.string().uuid({ message: 'Invalid Case ID format' }),
})
```

### 3.3 `src/features/documents/actions.ts`
```typescript
'use server'

import { revalidatePath } from 'next/cache'
import { createServerClient } from '@/lib/supabase'
import { UploadDocumentSchema, DeleteDocumentSchema, GetDocumentsSchema } from './validation'
import type { Document, UploadResponse, DeleteResponse, GetDocumentsResponse } from './types'

/**
 * Server Action: uploadDocument
 * Uploads a document file to the private 'documents' bucket and registers it via database RPC.
 */
export async function uploadDocument(formData: FormData): Promise<UploadResponse> {
  const supabase = createServerClient()
  let uploadedFilePath: string | null = null

  try {
    const caseId = formData.get('caseId')
    const type = formData.get('type')
    const file = formData.get('file')

    // 1. Zod input validation
    const parsed = UploadDocumentSchema.safeParse({ caseId, type, file })
    if (!parsed.success) {
      return {
        success: false,
        error: parsed.error.issues[0].message,
      }
    }

    const { caseId: validatedCaseId, type: validatedType, file: validatedFile } = parsed.data

    // 2. Authentication check
    const { data: { user }, error: authError } = await supabase.auth.getUser()
    if (authError || !user) {
      return {
        success: false,
        error: 'Unauthorized: Authenticated user session required',
      }
    }

    // 3. Setup IDs and Storage path
    const documentId = crypto.randomUUID()
    const filePath = `cases/${validatedCaseId}/${documentId}`
    uploadedFilePath = filePath

    // 4. Convert File to ArrayBuffer
    const fileBuffer = await validatedFile.arrayBuffer()

    // 5. Upload object to Storage (private documents bucket)
    const { error: storageError } = await supabase.storage
      .from('documents')
      .upload(filePath, fileBuffer, {
        contentType: validatedFile.type,
        upsert: false,
      })

    if (storageError) {
      return {
        success: false,
        error: `Storage upload failed: ${storageError.message}`,
      }
    }

    // 6. DB RPC invocation (creates record & activity log in a transaction)
    const { data: recordId, error: rpcError } = await supabase.rpc('upload_document_record', {
      p_case_id: validatedCaseId,
      p_type: validatedType,
      p_file_path: filePath,
      p_file_name: validatedFile.name,
      p_file_size: validatedFile.size,
      p_mime_type: validatedFile.type,
    })

    if (rpcError) {
      // Rollback: delete storage object on DB insertion failure
      await supabase.storage.from('documents').remove([filePath])
      return {
        success: false,
        error: `Database registration failed: ${rpcError.message}`,
      }
    }

    // 7. Retrieve full document row for response
    const { data: documentRow, error: fetchError } = await supabase
      .from('documents')
      .select('*')
      .eq('id', recordId)
      .single()

    if (fetchError || !documentRow) {
      return {
        success: true,
        data: {
          id: recordId,
          case_id: validatedCaseId,
          type: validatedType,
          file_path: filePath,
          file_name: validatedFile.name,
          file_size: validatedFile.size,
          mime_type: validatedFile.type,
          status: 'uploaded',
          uploaded_by: user.id,
          created_at: new Date().toISOString(),
          updated_at: new Date().toISOString(),
        } as Document,
        error: 'Document uploaded but failed to fetch complete database record.',
      }
    }

    // 8. Revalidate Cache
    revalidatePath(`/cases/${validatedCaseId}`)

    return {
      success: true,
      data: documentRow,
    }

  } catch (error: any) {
    return {
      success: false,
      error: error.message || 'An unexpected error occurred during document upload',
    }
  }
}

/**
 * Server Action: deleteDocument
 * Deletes a document record and its storage object atomically using the delete_document RPC.
 */
export async function deleteDocument(documentId: string): Promise<DeleteResponse> {
  const supabase = createServerClient()

  try {
    // 1. Zod input validation
    const parsed = DeleteDocumentSchema.safeParse({ documentId })
    if (!parsed.success) {
      return {
        success: false,
        error: parsed.error.issues[0].message,
      }
    }

    const { documentId: validatedDocumentId } = parsed.data

    // 2. Authentication check
    const { data: { user }, error: authError } = await supabase.auth.getUser()
    if (authError || !user) {
      return {
        success: false,
        error: 'Unauthorized: Authenticated user session required',
      }
    }

    // 3. Pre-fetch document's case context to support revalidation
    const { data: documentRow, error: fetchError } = await supabase
      .from('documents')
      .select('case_id')
      .eq('id', validatedDocumentId)
      .single()

    if (fetchError || !documentRow) {
      return {
        success: false,
        error: `Document not found or inaccessible: ${fetchError?.message || 'Not Found'}`,
      }
    }

    // 4. Execute atomic deletion RPC (handles storage and DB tables)
    const { error: rpcError } = await supabase.rpc('delete_document', {
      p_document_id: validatedDocumentId,
    })

    if (rpcError) {
      return {
        success: false,
        error: `Database deletion failed: ${rpcError.message}`,
      }
    }

    // 5. Revalidate cache
    revalidatePath(`/cases/${documentRow.case_id}`)

    return {
      success: true,
    }

  } catch (error: any) {
    return {
      success: false,
      error: error.message || 'An unexpected error occurred during document deletion',
    }
  }
}

/**
 * Server Action: getDocumentsByCase
 * Retrieves all documents registered under a specific case ID.
 */
export async function getDocumentsByCase(caseId: string): Promise<GetDocumentsResponse> {
  const supabase = createServerClient()

  try {
    // 1. Zod input validation
    const parsed = GetDocumentsSchema.safeParse({ caseId })
    if (!parsed.success) {
      return {
        success: false,
        error: parsed.error.issues[0].message,
      }
    }

    const { caseId: validatedCaseId } = parsed.data

    // 2. Authentication check
    const { data: { user }, error: authError } = await supabase.auth.getUser()
    if (authError || !user) {
      return {
        success: false,
        error: 'Unauthorized: Authenticated user session required',
      }
    }

    // 3. Retrieve database records
    const { data: documents, error: selectError } = await supabase
      .from('documents')
      .select('*')
      .eq('case_id', validatedCaseId)
      .order('created_at', { ascending: false })

    if (selectError) {
      return {
        success: false,
        error: `Failed to retrieve documents: ${selectError.message}`,
      }
    }

    return {
      success: true,
      data: documents || [],
    }

  } catch (error: any) {
    return {
      success: false,
      error: error.message || 'An unexpected error occurred during document retrieval',
    }
  }
}
```

## 4. Key Design Details & Rationale

### 1. Unified Return Signatures (No Raw Exception Propagation)
Client-facing server actions are designed to catch all database and runtime errors internally, wrapping them in a standard response object: `{ success: boolean; data?: any; error?: string }`.
- **Reasoning**: This conforms to standard API design and makes UI state handling (e.g. displaying error toast notifications in the browser) predictable and straightforward, eliminating unhandled promise rejections on the client.

### 2. Transaction Safety & Cleanup Rollback
In `uploadDocument`, files are uploaded to storage *before* calling the database RPC `upload_document_record`. If the RPC fails (e.g., due to database constraint violation like an invalid Case ID or user permissions), the action triggers a rollback deletion (`supabase.storage.from('documents').remove([filePath])`).
- **Reasoning**: Supabase Storage operations do not participate in Postgres SQL transactions. Performing manual rollback prevents the accumulation of orphaned files in private storage.

### 3. Server-Side Session and Integrity Audits
- **Reasoning**: The design uses `supabase.auth.getUser()` to authenticate users in the action, rather than `supabase.auth.getSession()` or passing a `userId` parameter. `getUser` validates the JWT signature with the Supabase auth server, guaranteeing that the session is valid and prevents spoofing of uploader IDs from client state.

### 4. Cache Revalidation and Optimistic UI Support
Each modification action (`uploadDocument`, `deleteDocument`) calls Next.js's `revalidatePath('/cases/{caseId}')`.
- **Reasoning**: Revalidation purges Next.js's Client Component cache and Server Component cache for that route, forcing Next.js to fetch the updated document list upon the next render. This guarantees consistent UI display without requiring manual browser refetches.
